## 1. 核心开发流程

### 第一步：同步与创建
在开始任何开发前，先同步远程 master 代码：
```bash
git checkout master
git pull origin master
git checkout -b feature/功能名称

```

### 第二步：本地开发与提交

开发过程中，保持小步快跑，多次提交：

```bash
git add .
# commit 信息请简述改动内容
# 采用 Conventional Commits 格式："类型(模块): 描述具体变动"
git commit -m "feat(auth): 增加用户登录接口逻辑" 

```

### 第三步：推送与申请 PR

```bash
# 推送本地分支到远程
git push origin feature/功能名称

```

推送后，在 GitHub/GitLab 网页端发起 **Pull Request (PR)**，目标分支指向 `master`。

---

## 2. Review 与 Approval 机制

**为了保证 master 分支永远是可运行的，所有 PR 必须经过评审。**

### 评审流程

1. **指定 Reviewer**：发起 PR 后，在右侧 Assignees 或 Reviewers 处勾选组内其他成员。
2. **代码评审 (Code Review)**：评审人查看代码逻辑、注释以及是否有明显的 Bug。
3. **发表评论**：评审人可以在特定代码行下留言提出疑问或修改建议。
4. **通过批准 (Approve)**：
* 评审人确认代码无误后，点击 **"Review changes"** 按钮。
* 选择 **"Approve"** 并填写类似 "LGTM" (Looks Good To Me) 的评语。
* **强制要求**：PR 必须获得 **至少 1 个 Approval** 才能执行 Merge。
5. **执行合并**：点击 **Merge** 来合并。
